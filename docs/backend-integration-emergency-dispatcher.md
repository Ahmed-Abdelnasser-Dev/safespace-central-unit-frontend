# Emergency Dispatcher — Backend Integration Spec

> **Audience:** Backend engineers wiring up the Emergency Dispatcher feature.  
> **Frontend location:** `src/features/emergencyDispatcher/`  
> **Routes (frontend):** `/cases` (console) · `/cases/:caseType/:caseId` (detail)  
> **Auth:** Cookie-based. Role gate = `emergency_dispatcher` or `admin`.

---

## 1. Architecture Overview

The Emergency Dispatcher is a real-time operations console. Dispatchers monitor two streams of incoming cases simultaneously:

- **SOS cases** — triggered by a victim via the SafeSpace mobile app.
- **Incident cases** — auto-detected by AI monitoring nodes on the highway network.

The console is a three-zone layout:
1. **Left** — Live case queue, sorted by severity then age.
2. **Center** — Interactive MapLibre map showing case pins and unit positions.
3. **Right** — Units roster showing all emergency units and their statuses.

Clicking a case opens a full **Case Detail Page** with victim/incident info, a dispatch panel to send units, and an assignment tracker.

All frontend state is currently managed by a local React reducer (`DispatcherProvider`). The integration swap path is: replace the mock data + reducer with API calls + Socket.IO events. The context shape (what the provider exposes) stays constant — only the data source changes.

## 2. Data Models

### 2.1 Case

A `Case` is a polymorphic object with a `caseType` discriminator. All shared fields are listed first, followed by type-specific fields.

```typescript
// Shared fields (both SOS and Incident)
interface BaseCase {
  id: string;                    // UUID
  caseType: 'sos' | 'incident';
  severity: 'HIGH' | 'MEDIUM' | 'LOW';
  status: CaseStatus;
  latitude: number;              // WGS-84 decimal degrees
  longitude: number;
  receivedAt: string;            // ISO 8601
  isUnread: boolean;             // per-dispatcher read flag (see §5.1)
  notes: Note[];
  assignmentIds: string[];       // IDs of Assignment records for this case
  attachments: Attachment[];
  assignedDispatcherId: string | null;  // dispatcher assigned to own this case
}

type CaseStatus =
  | 'queued'       // arrived, no dispatcher has opened it
  | 'acknowledged' // a dispatcher has opened it (auto-set on first view)
  | 'active'       // at least one unit dispatched
  | 'escalated'    // flagged for supervisor
  | 'resolved'     // response confirmed successful
  | 'false_alarm'  // incident detection was wrong (incident cases only)
  | 'closed';      // manually closed by dispatcher
```

#### SOS Case (caseType = 'sos')

```typescript
interface SOSCase extends BaseCase {
  caseType: 'sos';
  emergencyType: 'ROAD_ACCIDENT' | 'MEDICAL' | 'FIRE' | 'POLICE' | 'UNSPECIFIED';
  victim: VictimProfile;
  medicalProfile: MedicalProfile | null;
  emergencyContacts: EmergencyContact[];
}

interface VictimProfile {
  fullName: string;
  nationalId: string | null;   // Egyptian national ID (14 digits)
  dob: string | null;          // ISO 8601 date (YYYY-MM-DD)
  phone: string | null;        // E.164 format e.g. "+201012345678"
}

interface MedicalProfile {
  bloodType: string | null;    // e.g. "O+", "A-"
  conditions: string[];        // e.g. ["Hypertension", "Diabetes"]
  medications: string[];       // e.g. ["Lisinopril"]
  allergies: string[];         // e.g. ["Penicillin"]
  notes: string | null;
}

interface EmergencyContact {
  name: string;
  relationship: string;        // e.g. "Spouse", "Parent"
  phone: string;               // E.164
}
```

#### Incident Case (caseType = 'incident')

```typescript
interface IncidentCase extends BaseCase {
  caseType: 'incident';
  incidentType: 'COLLISION' | 'STOPPED_VEHICLE' | 'ROAD_HAZARD' | 'UNSPECIFIED';
  confidence: number;          // AI confidence score 0.0–1.0
  affectedLanes: string[];     // e.g. ["L1", "L2"]
  nodeLabel: string;           // human-readable node/location, e.g. "Highway Node 07 — Km 42 Ismailia"
}
```

### 2.2 Note

```typescript
interface Note {
  id: string;
  caseId: string;
  authorType: 'system' | 'dispatcher';
  authorId: string | null;     // dispatcher user ID when authorType = 'dispatcher'; null for system
  content: string;
  createdAt: string;           // ISO 8601
}
```

**System notes** are auto-generated by the backend on state transitions (case received, unit dispatched, status changed, etc.). **Dispatcher notes** are written by the operator from the UI.

---

### 2.3 Attachment

```typescript
interface Attachment {
  id: string;
  type: 'image' | 'video';
  url: string;                 // CDN URL — must be publicly accessible
  caption: string | null;
  uploadedAt: string;          // ISO 8601
}
```

For SOS cases, attachments are photos/videos sent by the victim via the mobile app. For incident cases, they are frames captured by the highway AI node at detection time.

---

### 2.4 Unit (Emergency Unit)

```typescript
interface Unit {
  id: string;
  name: string;                // e.g. "Ambulance Unit 3"
  unitType: 'ambulance' | 'police' | 'fire' | 'civil_protection';
  status: UnitStatus;
  stationId: string;           // FK → Station
  homeBase: {
    name: string;
    latitude: number;
    longitude: number;
  };
  currentLatitude: number;     // live GPS position
  currentLongitude: number;
  lastLocationAt: string;      // ISO 8601 — when GPS was last updated
}

type UnitStatus = 'available' | 'en_route' | 'on_scene' | 'off_duty';
```

The frontend **filters out `off_duty` units** from the dispatch panel but still shows them in the roster with reduced opacity.

---

### 2.5 Station

```typescript
interface Station {
  id: string;
  name: string;                // e.g. "Ismailia Central Hospital"
  shortName: string;           // abbreviated, used on map labels
  stationType: 'ambulance' | 'police' | 'fire' | 'civil_protection';
  latitude: number;
  longitude: number;
}
```

Stations are rendered as pinned markers on the console map. They are mostly static; changes are rare.

---

### 2.6 Assignment

An Assignment links one Unit to one Case for a specific response.

```typescript
interface Assignment {
  id: string;
  caseId: string;
  unitId: string;
  status: AssignmentStatus;
  dispatchedAt: string;        // ISO 8601
  assignedBy: string;          // dispatcher user ID
}

type AssignmentStatus = 'notified' | 'en_route' | 'on_scene' | 'completed' | 'cancelled';
```

**Assignment lifecycle (forward only):**

```
notified → en_route → on_scene → completed
                                         ↑
(can also transition from notified/en_route to) cancelled
```

The frontend enforces one-step-at-a-time advancement. The backend should validate that the requested transition is valid.

---

### 2.7 Dispatcher (Current User)

The logged-in dispatcher's profile is already available via the existing auth session (`state.auth.user`). The dispatcher feature only needs `id` and `name` from it. The `shiftStart` field is useful for display purposes.

```typescript
interface DispatcherSession {
  id: string;
  name: string;
  shiftStart: string;          // ISO 8601 — when the dispatcher clocked in
}
```

## 3. REST API Endpoints

**Base path:** `/api/dispatcher`  
**Auth:** All endpoints require a valid session cookie. Non-dispatcher roles receive `403`.

---

### 3.1 Cases

#### `GET /api/dispatcher/cases`

Returns all cases visible to this dispatcher. Defaults to active cases; supports filtering.

**Query params:**

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `status` | comma-separated `CaseStatus` | `queued,acknowledged,active,escalated` | Filter by status. Pass `all` for no filter. |
| `caseType` | `sos` \| `incident` | — | Optional type filter |
| `page` | integer | `1` | |
| `limit` | integer | `50` | |

**Response `200`:**

```json
{
  "data": [Case],
  "meta": {
    "total": 42,
    "page": 1,
    "limit": 50
  }
}
```

Notes are **not** embedded in the list response (to keep payloads small). They are fetched in the single-case endpoint. `attachments` and `assignmentIds` are included.

---

#### `GET /api/dispatcher/cases/:id`

Returns a single case with full detail including embedded `notes` array.

**Response `200`:**

```json
{
  "data": Case   // full object including notes[]
}
```

**Response `404`:** case not found.

---

#### `PATCH /api/dispatcher/cases/:id/acknowledge`

Called automatically when a dispatcher opens a case. Transitions status from `queued` → `acknowledged` and clears `isUnread` for this dispatcher. Idempotent — if already acknowledged or further along, returns the current state unchanged.

**Request body:** none  
**Response `200`:** `{ "data": Case }`

---

#### `PATCH /api/dispatcher/cases/:id/status`

Dispatcher-initiated status changes. Only certain transitions are allowed per current status.

**Request body:**

```json
{
  "status": "escalated" | "resolved" | "false_alarm" | "closed"
}
```

**Allowed transitions:**

| From | To (allowed) |
|------|-------------|
| `queued` \| `acknowledged` \| `active` | `escalated`, `resolved`, `false_alarm`, `closed` |
| `escalated` | `resolved`, `false_alarm`, `closed` |
| `resolved` \| `false_alarm` \| `closed` | _(terminal — no transitions)_ |

**Side effects (backend must handle):**
- `resolved`: All active assignments (`notified`/`en_route`/`on_scene`) → `completed`; their units → `available`. System note appended.
- `false_alarm`: All active assignments → `cancelled`; their units → `available`. System note appended.
- `closed`: All active assignments → `completed`; their units → `available`. System note appended.
- `escalated`: System note appended.

**Response `200`:** `{ "data": Case }` (updated, with appended system note)  
**Response `422`:** invalid transition with error detail.

---

#### `POST /api/dispatcher/cases/:id/notes`

Adds a dispatcher note to a case.

**Request body:**

```json
{
  "content": "Victim confirmed location, heading to extraction point."
}
```

**Response `201`:**

```json
{
  "data": Note   // the newly created note
}
```

The frontend appends the returned note to the case's `notes` array without refetching the whole case.

---

#### `POST /api/dispatcher/cases/:id/assign-dispatcher`

Assigns this case to a specific dispatcher. Used internally when the backend routes a case to a dispatcher ("assigned to me" feature). Can also be called by the dispatcher themselves.

**Request body:**

```json
{
  "dispatcherId": "user-uuid"
}
```

**Response `200`:** `{ "data": Case }`

---

### 3.2 Assignments

#### `POST /api/dispatcher/cases/:id/assignments`

Dispatches one or more units to a case. Creates one Assignment per unit.

**Request body:**

```json
{
  "unitIds": ["unit-amb-3", "unit-police-7"]
}
```

**Side effects (backend must handle):**
- Each unit's `status` → `en_route`.
- Case `status` → `active` (if currently `queued`, `acknowledged`, or `escalated`).
- System note appended: `"{unit names} dispatched"`.
- `assignmentIds` on the case updated.

**Response `201`:**

```json
{
  "data": [Assignment]   // array of created assignments
}
```

**Response `422`:** if any `unitId` is not `available` or is already actively assigned to this case.

---

#### `PATCH /api/dispatcher/assignments/:assignmentId/status`

Advances an assignment through its lifecycle.

**Request body:**

```json
{
  "status": "en_route" | "on_scene" | "completed"
}
```

**Validation:** Only the next step in the forward chain is accepted:
- `notified` → `en_route`
- `en_route` → `on_scene`
- `on_scene` → `completed`

**Side effects:**
- `completed`: unit's `status` → `available`.

**Response `200`:** `{ "data": Assignment }`  
**Response `422`:** invalid transition.

---

#### `PATCH /api/dispatcher/assignments/:assignmentId/cancel`

Cancels an in-progress assignment.

**Allowed from:** `notified`, `en_route`  
**Side effects:** unit's `status` → `available`.

**Response `200`:** `{ "data": Assignment }`

---

### 3.3 Units

#### `GET /api/dispatcher/units`

Returns all emergency units with current GPS position and status.

**Response `200`:**

```json
{
  "data": [Unit]
}
```

This endpoint is polled or replaced by real-time push (see §4). The frontend renders all units on the map and in the roster panel.

---

#### `GET /api/dispatcher/units/nearest`

Returns units ranked by straight-line distance (haversine) from a given coordinate. The frontend currently computes this client-side from the full unit list, but the backend should offer this endpoint for future scalability.

**Query params:**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `lat` | float | yes | Incident/SOS latitude |
| `lng` | float | yes | Incident/SOS longitude |
| `excludeOffDuty` | boolean | no | Default `true` |
| `limit` | integer | no | Default `20` |

**Response `200`:**

```json
{
  "data": [
    { ...Unit, "distanceKm": 2.4 }
  ]
}
```

---

### 3.4 Stations

#### `GET /api/dispatcher/stations`

**Response `200`:**

```json

  "data": [Station]
```

## 4. Real-Time Events (Socket.IO)

All real-time events go over the existing Socket.IO connection (`src/services/socketService.js`). The dispatcher feature should subscribe to a dedicated namespace or room.

**Suggested room:** `dispatcher:{dispatcherId}` for personal events + `dispatcher:global` for broadcast events.

---

### 4.1 Events the Server Emits (Frontend Listens)

#### `case:new`
A new case has arrived (SOS from mobile app or incident from AI node).

```json
{
  "event": "case:new",
  "data": Case    // full case object, status = 'queued', isUnread = true
}
```

The frontend prepends this to the case list and increments the unread badge on the relevant tab.

---

#### `case:updated`
A case's fields have changed (status, assignedDispatcherId, etc.).

```json
{
  "event": "case:updated",
  "data": Case    // full updated case object
}
```

Used for cross-dispatcher consistency — if dispatcher A resolves a case, dispatcher B's console reflects it immediately.

---

#### `unit:location`
One or more unit GPS positions have updated. Emitted frequently (e.g. every 5s per active unit).

```json
{
  "event": "unit:location",
  "data": {
    "unitId": "unit-amb-3",
    "latitude": 30.601,
    "longitude": 32.271,
    "lastLocationAt": "2026-06-17T10:30:00Z"
  }
}
```

---

#### `unit:status`
A unit's operational status has changed.

```json
{
  "event": "unit:status",
  "data": {
    "unitId": "unit-amb-3",
    "status": "en_route"
  }
}
```

---

#### `assignment:updated`
An assignment's status has changed (could be triggered by a field unit's mobile app or another dispatcher).

```json
{
  "event": "assignment:updated",
  "data": Assignment    // full updated assignment
}
```

---

#### `dispatcher:assigned`
A case has been routed to this dispatcher ("assigned to me" notification banner).

```json
{
  "event": "dispatcher:assigned",
  "data": {
    "caseId": "case-sos-1"
  }
}
```

The frontend shows a `NewAssignmentModal` alert when this fires. Only emitted to the target dispatcher's room.

---

### 4.2 Events the Client Emits (Server Listens)

The frontend currently performs all mutations via REST (see §3). WebSocket emits from the client are not required for the initial integration. If the team wants to avoid HTTP round-trips for high-frequency updates (unit location from field units' apps), those should be separate mobile-side concerns.

## 5. Business Logic Reference

### 5.1 `isUnread` Flag

`isUnread` is a **per-dispatcher** field. When a dispatcher opens a case (`PATCH /cases/:id/acknowledge`), `isUnread` is cleared **for that dispatcher only**. Other dispatchers still see it as unread until they open it.

Implementation options:
- Separate `case_reads` join table: `(case_id, dispatcher_id, read_at)`.
- The `isUnread` flag in the Case response is computed relative to the requesting dispatcher.

---

### 5.2 Default Dispatch Sets

When a case detail page loads, the frontend **automatically pre-selects** the nearest available unit of each required type. The required types per case/emergency type are:

| Case Type | Emergency/Incident Type | Required Unit Types |
|-----------|------------------------|---------------------|
| SOS | `MEDICAL` | ambulance |
| SOS | `FIRE` | ambulance, fire |
| SOS | `POLICE` | ambulance, police |
| SOS | `ROAD_ACCIDENT` | ambulance, police, civil_protection |
| SOS | `UNSPECIFIED` | ambulance, police, civil_protection |
| Incident | `COLLISION` | ambulance, police, civil_protection |
| Incident | `STOPPED_VEHICLE` | police |
| Incident | `ROAD_HAZARD` | police, civil_protection |
| Incident | `UNSPECIFIED` | ambulance, police, civil_protection |

The `GET /api/dispatcher/units/nearest` endpoint can serve this pre-selection server-side if the client passes `caseId` and the backend knows the case type.

---

### 5.3 Case Status Flow

```
                     ┌──────────┐
   SOS/Incident ───► │  queued  │
   arrives           └────┬─────┘
                          │ dispatcher opens case
                     ┌────▼──────────┐
                     │ acknowledged  │
                     └────┬──────────┘
                          │ units dispatched
                     ┌────▼─────┐
                     │  active  │◄─────────────────────┐
                     └────┬─────┘                      │
                          │                            │ (additional
                    ┌─────┼──────────────┐             │  dispatches)
                    │     │              │             │
            escalate│     │ resolve    false           │
                    ▼     ▼    alarm      ▼            │
               escalated  resolved  false_alarm  closed│
                    │                                  │
                    └──────────────────────────────────┘
                         (all terminal statuses)
```

---

### 5.4 Unit Status Transitions

Unit status is driven by assignment events:

| Trigger | Unit Status Change |
|---------|--------------------|
| Unit dispatched to case | → `en_route` |
| Assignment cancelled | → `available` |
| Assignment completed (or case resolved/closed) | → `available` |
| Field unit marks themselves on scene | → `on_scene` (driven by `assignment:updated`) |

Units with status `off_duty` are shown in the roster but excluded from the dispatch panel and nearest-units ranking.

---

### 5.5 ETA Estimation

The frontend estimates ETA using a simplified formula:

```
eta_minutes = max(1, round((distance_km / 50) * 60))
```

This assumes a constant 50 km/h average speed. No routing service is integrated. The backend can return a more accurate ETA from a routing API if available — add an optional `etaMinutes` field to the `Unit` response in `GET /units/nearest`.

## 6. Error Response Format

All endpoints should follow the existing API envelope:

```json
{
  "success": false,
  "data": null,
  "error": "Human-readable error message"
}
```

HTTP status codes:

| Code | Meaning |
|------|---------|
| `200` | Success |
| `201` | Created |
| `400` | Bad request / validation error |
| `401` | Unauthenticated |
| `403` | Forbidden (wrong role) |
| `404` | Resource not found |
| `422` | Invalid state transition |
| `500` | Server error |
